# AST3 — Дорожная карта реализации

> **AST3** (Army Style Task & Time Tracking) — внутренняя платформа управления задачами, временем и проектами для закрытого контура с требованиями ФСТЭК.
> Лицензия проекта: **MIT**. Стек: **Django + Strawberry GraphQL + PostgreSQL (Postgres Pro Certified) + Refine/React + Astra Linux**.

---

## 0. Аналитическая сводка

### 0.1. Что мы строим
Закрытая on-premise платформа на 30–100 пользователей, объединяющая:
- Канбан-доски и трекинг задач (ПО + производство экспериментальных изделий);
- Учёт рабочего времени (тайм-трекинг);
- Календарь проектов (Гант);
- Аналитику прогресса по людям/подразделениям/проектам;
- Автоматический расчёт рисков срыва сроков;
- Полный аудит действий;
- Реестр отложенных работ (бэклог);
- Массовый импорт/экспорт задач.

### 0.2. Ключевые ограничения
| Ограничение | Следствие |
|---|---|
| ФСТЭК (предположительно КС1/КС2) | Стек только из реестра, свой код, обязательный SDLC с SAST/DAST |
| Только MIT-лицензии зависимостей | Plane/OpenProject/Taiga отпадают как код-база; используем как референс |
| Закрытый контур | Никаких SaaS-зависимостей, оффлайн-установка пакетов, локальные зеркала |
| Свой сертификат на изделие | Версионирование, документация ИБ, формализованный процесс сборки |

### 0.3. Архитектурные принципы
1. **Monorepo** — `backend/` + `client/` + `infra/` + `docs/` в одном репозитории.
2. **API-first** — GraphQL как единственная точка входа для клиента; REST только для системных нужд (healthz, metrics, file upload через tus).
3. **Domain-driven** — разбиение на bounded contexts: `accounts`, `projects`, `tasks`, `tracking`, `risks`, `audit`, `reports`, `integrations`.
4. **Event-driven для фоновой работы** — Celery + Redis (или RabbitMQ, если Redis не пройдёт по ФСТЭК).
5. **RBAC + ABAC** — роли + контекст подразделения/проекта; всё через единый `permissions`-слой.
6. **Audit by design** — каждое мутирующее действие пишется в неизменяемый журнал.
7. **Security by default** — JWT с короткоживущим access + refresh-rotation, CSP, HSTS, защита от IDOR на уровне резолверов.

## 2. Детальный план (актуальное состояние)

### 🔄 Фаза 3. Риски и аналитика для руководства (в процессе)

#### 3.1. Модуль рисков
- [ ] Модель `risks.Risk` (ручные риски: `title`, `description`, `level`, `probability`, `impact`, `owner`, `mitigation`, `status`).
- [ ] Авто-риски на задаче: считаются периодической Celery-задачей.
- [ ] Правила (вынести в конфиг/БД):
  - `planned_end` в прошлом и статус ≠ done → **high**;
  - срок до `planned_end` ≤ 20% оставшегося плана и `progress` < ожидаемого → **medium**;
  - блокирующие зависимости не закрыты при близком сроке → **high**;
  - перерасход оценки > 50% → **medium**.
- [ ] Журнал изменений уровня риска (для аудита).
- [ ] Уведомления: email + внутренние нотификации.

#### 3.2. Дашборды
- [ ] Дашборд руководителя: риск-хитмап по проектам, топ-5 проблемных задач.
- [ ] Дашборд PM проекта: burndown, распределение задач по статусам.
- [ ] Дашборд сотрудника: мои задачи, моё время за неделю/месяц.
- [ ] Графики через **Recharts** (MIT).
- [ ] Все агрегации — на уровне БД (materialized views или CTE).

#### 3.3. Отчёты
- [ ] Конструктор отчётов (MVP — преднастроенные шаблоны):
  - "Прогресс по подразделениям за период";
  - "Реестр рисков";
  - "Трудозатраты по проекту";
  - "Просроченные задачи".
- [ ] Экспорт в XLSX и PDF.
- [ ] Расписание отчётов: еженедельная рассылка руководителю.

**Definition of Done для Ф3:** руководитель видит дашборд с рисками и прогнозом по портфелю.

## 3. Главная задача: импорт и экспорт задач

> **Цель:** массовое создание и выгрузка задач без ручного ввода через UI.

### 3.1. Импорт задач
- [ ] Разработка GraphQL mutation `importTasks(projectId, format, file)`:
  - Поддержка форматов: **XLSX**, **CSV**, **JSON**.
  - Валидация полей на стороне сервера (типы, коды проектов, справочники статусов/приоритетов).
  - Пакетная запись с сохранением чанками по 500 записей.
  - Строгая типизация: каждый импортируемый столбец сопоставляется с моделью; неизвестные поля — в `metadata` или отклонение.
- [ ] Фоновая Celery-задача для больших файлов (>10 МБ / >5000 строк).
- [ ] Умное сопоставление (fuzzy-match): создание отсутствующих статусов/исполнителей или предложение ручного выбора.
- [ ] Предпросмотр перед подтверждением: таблица с diff (новые / дубликаты / ошибки).
- [ ] Полный аудит: кто, когда, из какого файла, сколько создано/обновлено/пропущено.

### 3.2. Экспорт задач
- [ ] Разработка GraphQL query `exportTasks(projectId, filter, format)`:
  - Поддержка форматов: **XLSX** (с таблицей стилей), **CSV**, **JSON**.
  - Фильтры: по статусу, исполнителю, сроку, тегу, приоритету.
  - Выгрузка вложений: отдельный архив или ссылки в виде относительных путей.
- [ ] Ограничение по размеру: сотни тысяч строк (стриминг через `xlsxwriter`/`openpyxl`).
- [ ] Кодировка и локаль: UTF-8, формат дат `DD.MM.YYYY`, разделитель `;` или `,` по выбору.
- [ ] Шаблоны экспорта: пользователь сохраняет набор столбцов/фильтров.

### 3.3. UI/UX
- [ ] Страница "Импорт / Экспорт" в карточке проекта.
- [ ] Drag-and-drop загрузки файла, превью первой 10 строк.
- [ ] Прогресс-бар для фоновой обработки.
- [ ] Скачивание готового файла через подписанную ссылку.

### 3.4. Безопасность и ограничения
- [ ] Лимит размера файла (настраиваемо, по-умолчанию 50 МБ).
- [ ] Проверка MIME-типа и расширения на уровне HTTP + GraphQL.
- [ ] Разрешение только для ролей: `manager`, `admin` (исключить `viewer`).
- [ ] Валидация макросов в XLSX (запрет на внедрение формул).

**Definition of Done для импорта/экспорта:** пользователь загружает XLSX/CSV/JSON → система валидирует → показывает diff → массово создаёт/обновляет задачи → пишет AuditLog; аналогично для выгрузки.

## 4. Реестр отложенных работ (бэклог)

> **Цель:** централизованный реестр идей, улучшений, дефектов и инициатив с управлением статусами до их превращения в задачи.

### 4.1. Модель данных
- [ ] Модель `projects.BacklogItem` (или верхнеуровневая `tasks.BacklogRegistry`):
  - `title`, `description`, `source` (`idea|bug|improvement|requirement|other`).
  - `status`: `draft|review|approved|rejected|planned|in_progress|done`.
  - `priority`, `effort_estimate` (story points или часы).
  - `tags`, `related_project`, `related_task`.
  - `author`, `assignee` (создатель, инициатор).
  - `created_at`, `updated_at`, `decided_at`.
- [ ] Инварианты:
  - Только `approved` и `planned` transitioning to tasks.
  - Через `AuditLog` на каждое изменение статуса.

### 4.2. GraphQL API
- [ ] Query `backlogItems(filter, pagination)`.
- [ ] Mutations:
  - `createBacklogItem`, `updateBacklogItem`, `deleteBacklogItem`.
  - `changeBacklogStatus(itemId, status, comment)` — обязательный комментарий/причина перехода.
- [ ] Подписки на изменения статуса для автора и ответственного.

### 4.3. UI
- [ ] Раздел "Реестр отложенных работ" в сайдбаре.
- [ ] Kanban-доска со столбцами по статусам бэклога.
- [ ] Детальная карточка с историей переходов.
- [ ] Действия: "Создать задачу из пункта бэклога" (преобразование с заполнением полей `Task` из бэклога).

### 4.4. Процессы
- [ ] Еженедельное совещание: перенос из `approved` → `planned`.
- [ ] Уведомление инициатору о переходе в `in_progress` и `done`.
- [ ] SLA: не одобренные пункты старше 30 дней — авто-уведомление ответственному.

## 5. Улучшенная диаграмма Ганта (календарь проектов)

> **Цель:** отображение задач по времени с учётом зависимостей, критического пути и ресурсов.

### 5.1. Данные
- [ ] Query `ganttData(projectId, range, granularity)`:
  - Вернёт задачи с `planned_start`, `planned_end`, `progress`.
  - Вернёт зависимости `TaskDependency` с типами FS/SS/FF/SF и `lag`.
  - Вычисленный критический путь (CPM) и маргиналы.
- [ ] Поддержка zoom-уровней: день / неделя / месяц / квартал.
- [ ] Фильтрация: только мои задачи, по исполнителю, по статусу, по типу.

### 5.2. Визуализация
- [ ] Бар задачи: цвет по статусу/приоритету/риску; hover — tooltip с фактическими часами и прогрессом.
- [ ] Линии зависимостей: стрелки с правильным типом (FS/SS/FF/SF), подсветка циклических связей красным.
- [ ] Критический путь: жирная рамка или пунктир.
- [ ] Просроченные задачи: красная подсветка, blinking-индикатор.
- [ ] Текущая дата: вертикальная линия,Today".
- [ ] Drag-resize: изменение `planned_start`/`planned_end` мутацией `updateTaskSchedule`.
- [ ] Drag dependencies: перетаскивание конца стрелки → мутация `updateTaskDependency`.

### 5.3. Ресурсы и нагрузка
- [ ] Отображение загрузки исполнителя по дням (гистограмма или heatmap).
- [ ] Предупреждение при пересечении > 8 ч/день на одного человека.
- [ ] Группировка задач по исполнителю или по подразделению.

### 5.4. Печать и экспорт
- [ ] Экспорт в PNG/PDF для совещаний.
- [ ] Формат A3/A4, легенда, шкала времени, критические пути.

**Definition of Done для Ганта:** PM перетаскивает бар → сроки задачи и зависимостей обновляются в реальном времени; при перегрузке ресурса — предупреждение; при печати — читаемый PDF.
